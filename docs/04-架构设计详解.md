## 二、为什么选 Pipeline + 并行 + 循环？

### 2.1 三种编排模式对比

| 模式 | 定义 | 适用场景 | 复杂度 |
|------|------|---------|--------|
| **Pipeline** | 线性串联，A→B→C | 步骤有先后依赖 | 低 |
| **DAG** | 有向无环图，支持分叉和汇合 | 复杂依赖关系 | 中 |
| **State Machine** | 状态节点 + 条件转移 | 有循环、有分支 | 高 |

### 2.2 本项目的选择理由

我们的系统是 **Pipeline + DAG分叉 + 状态机循环** 的混合：

1. **串行部分（Pipeline）**: Preference → Destination 有严格的先后关系，必须先知道偏好才能推荐目的地
2. **并行部分（DAG分叉）**: Flight + Hotel + Activity 三个 Agent 互相独立，可以同时执行
3. **循环部分（State Machine）**: Budget 校验后可能需要回退调整，这是一个 while 循环

### 2.3 为什么不用纯 DAG？

纯 DAG 不支持循环。我们的预算调整需要"回退重做"，这在 DAG 中无法表达。
LangGraph 的解决方案是在 conditional_edge 中引入 cycle，本质上就是状态机。

### 2.4 为什么不用 LangGraph / CrewAI？

**面试答法**: 我在自研实现和框架之间做了权衡：
- 自研 Pipeline + ParallelExecutor + BudgetLoop：代码完全可控，面试可以深入讲任何细节
- LangGraph：适合更复杂的图结构，但本项目的流程相对线性，用框架反而过度设计
- 如果要上生产，会考虑迁移到 LangGraph，因为它提供了 checkpointing、human-in-the-loop 等企业级特性

## 三、6 个 Agent 的设计

### 3.1 为什么是 6 个而不是 3 个或 10 个？

**Agent 拆分原则**：
1. **单一职责**: 每个 Agent 只做一件事
2. **独立可测**: 每个 Agent 可以单独测试
3. **并行可能**: 互不依赖的 Agent 应该拆开以支持并行
4. **粒度适中**: 太细会增加通信开销，太粗会丧失并行优势

| Agent | 是否可以合并？ | 为什么不合并？ |
|-------|--------------|--------------|
| Preference + Destination | 不建议 | 偏好收集可能需要多轮对话，逻辑复杂度不同 |
| Flight + Hotel | 不建议 | 数据源不同（航班API vs 酒店API），且需要并行 |
| Activity + Hotel | 不建议 | 活动规划需要按天安排时间槽，逻辑更复杂 |
| Budget 独立 | 必须 | 预算校验是全局性的，需要汇总所有费用 |

### 3.2 Agent 基类设计（模板方法模式）

```python
class BaseAgent(ABC):
    async def run(self, state):      # 模板方法：不可覆盖
        log("开始执行")
        try:
            state = await self.execute(state)  # 调用子类实现
        except Exception as e:
            state.error_messages.append(str(e))
        return state

    @abstractmethod
    async def execute(self, state):  # 子类必须实现
        ...
```

**面试考点**：模板方法模式的好处是什么？
- 统一了日志、错误处理、指标收集
- 子类只关注业务逻辑
- 新增 Agent 只需要实现 execute() 方法

### 3.3 Agent 间通信方式：共享状态

本项目使用 **共享状态（TravelPlanState）** 模式：

```python
class TravelPlanState:
    preferences: UserPreferences      # Preference Agent 写入
    destination_rec: DestinationRec   # Destination Agent 写入
    flight_result: FlightSearchResult # Flight Agent 写入
    hotel_result: HotelSearchResult   # Hotel Agent 写入
    activity_result: ActivityResult   # Activity Agent 写入
    budget_breakdown: BudgetBreakdown # Budget Agent 写入
```

**为什么用共享状态而不是消息传递？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 共享状态 | 简单直观，并行Agent各写各的字段 | 需要注意并发写入 |
| 消息传递 | 解耦度高，Agent 无需知道彼此 | 实现复杂，需要序列化/反序列化 |
| Event Bus | 灵活，支持发布/订阅 | 调试困难，流程不易追踪 |

本项目选共享状态因为：
1. 并行 Agent 写入的是**不同字段**，不存在写冲突
2. Pipeline 流程线性，不需要复杂的事件机制
3. Pydantic 模型提供类型安全

## 四、并行执行设计

### 4.1 为什么并行？

```
串行: Flight(3s) → Hotel(2s) → Activity(2s) = 7s
并行: max(Flight(3s), Hotel(2s), Activity(2s)) = 3s
```

延迟降低: (7 - 3) / 7 = **57%**

### 4.2 Python 并行方案对比

| 方案 | 适用场景 | GIL 影响 |
|------|---------|---------|
| asyncio.gather | IO 密集（网络请求） | 不受影响 |
| ThreadPoolExecutor | IO 密集 + 部分 CPU | 受 GIL 限制 |
| ProcessPoolExecutor | CPU 密集 | 不受影响 |
| multiprocessing | CPU 密集，需要进程隔离 | 不受影响 |

本项目选 **asyncio.gather**，因为 Agent 的核心操作是网络 IO（调用 LLM API / 搜索 API）。

### 4.3 并行错误处理

```python
results = await asyncio.gather(*tasks, return_exceptions=True)
for agent, result in zip(agents, results):
    if isinstance(result, Exception):
        state.error_messages.append(f"{agent.name}: {result}")
```

**关键设计**: `return_exceptions=True` 让一个 Agent 失败不影响其他 Agent。
例如航班搜索超时，酒店和活动搜索仍然正常完成。

### 4.4 Java 并行实现

```java
CompletableFuture<Void> all = CompletableFuture.allOf(
    CompletableFuture.runAsync(() -> flightAgent.execute(state)),
    CompletableFuture.runAsync(() -> hotelAgent.execute(state)),
    CompletableFuture.runAsync(() -> activityAgent.execute(state))
);
all.join();
```

### 4.5 Go 并行实现

```go
var wg sync.WaitGroup
var mu sync.Mutex
wg.Add(3)
for _, agent := range []Agent{flight, hotel, activity} {
    go func(a Agent) {
        defer wg.Done()
        mu.Lock()
        a.Run(state)
        mu.Unlock()
    }(agent)
}
wg.Wait()
```

## 五、预算循环设计（重点！）

### 5.1 循环流程

```
初始搜索
  ↓
Budget 校验 ──通过──▶ 完成
  │
  │ 超预算
  ▼
第1轮: 砍活动费用 (最高40%)
  ↓
Budget 校验 ──通过──▶ 完成
  │
  │ 超预算
  ▼
第2轮: 降酒店等级 (最高35%)
  ↓
Budget 校验 ──通过──▶ 完成
  │
  │ 超预算
  ▼
第3轮: 换经济航班 (最高25%)
  ↓
Budget 校验 ──通过──▶ 完成
  │
  │ 仍超预算
  ▼
返回当前最优方案 + 警告
```

### 5.2 三个终止条件

1. **预算通过**: `total_cost <= budget` → 正常完成
2. **达到最大轮次**: `adjustment_round >= 3` → 返回当前最优方案
3. **不可恢复错误**: Agent 全部失败 → 报错

### 5.3 渐进式降级策略

| 轮次 | 调整对象 | 最大削减比例 | 理由 |
|------|---------|------------|------|
| 1 | 活动费用 | 40% | 活动弹性最大，免费景点很多 |
| 2 | 酒店等级 | 35% | 住宿是第二大开支，降级感知相对小 |
| 3 | 航班选择 | 25% | 航班选择空间最小，且影响体验最大 |

**为什么这个顺序？**
- 用户对活动降级感知最小（少一个收费景点 vs 住差酒店 vs 多一次转机）
- 渐进式降级保证每轮只动一个维度，避免震荡

### 5.4 防止无限循环的关键

```python
for attempt in range(self.max_retries + 1):  # 最多 4 次（初始 + 3轮调整）
    if state.state == PlanningState.COMPLETED:
        break
```

**面试追问**: "如果每轮调整幅度太小，3轮仍然不够怎么办？"

> 可以引入动态幅度计算：`cut_ratio = min(0.5, over_amount / category_cost)`。
> 超预算越多，削减越狠。极端情况下，第3轮可以直接推荐完全不同的低价目的地。

## 七、扩展性设计

### 如果要新增一个 Agent？

例如增加 **WeatherAgent**（天气预报Agent）：

1. 创建 `weather_agent.py`，继承 `BaseAgent`
2. 实现 `execute()` 方法
3. 在 `ParallelExecutor` 中注册：

```python
self.parallel_executor = ParallelExecutor(
    agents=[self.flight_agent, self.hotel_agent, self.activity_agent, self.weather_agent],
)
```

4. 在 `TravelPlanState` 中添加 `weather_result` 字段

**总共改动 4 个地方，不影响现有 Agent**。这就是良好架构设计的价值。

### 如果要替换 Mock 为真实 API？

只需要修改各 Agent 的 `execute()` 方法内部实现，不影响编排层。
这就是**依赖倒置原则**的体现：编排层依赖 Agent 接口，不依赖具体实现。
