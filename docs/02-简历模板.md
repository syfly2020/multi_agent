## 二、STAR 法则详解

### 什么是 STAR 法则？

STAR 是面试中描述项目经历的黄金框架：

| 维度 | 含义 | 占比 | 技巧 |
|------|------|------|------|
| **S**ituation | 项目背景与挑战 | 15% | 简短交代，突出痛点 |
| **T**ask | 你的具体任务 | 15% | 明确你的角色和目标 |
| **A**ction | 你采取的行动 | 50% | 技术细节，这是核心 |
| **R**esult | 成果与数据 | 20% | 量化！量化！量化！ |

### 本项目 STAR 完整示例

#### Situation（背景）

> 传统旅行社行程规划依赖人工，规划一个 5 天行程需要 8 小时以上，
> 而且无法实时比价和动态调整预算。随着 AI Agent 技术成熟，
> 我希望探索多 Agent 协作在实际业务场景的落地。

#### Task（任务）

> 我负责整个系统的架构设计和核心开发，目标是：
> 1. 设计并实现一个 6-Agent 协作系统
> 2. 支持并行搜索降低延迟
> 3. 实现自动预算控制循环
> 4. 提供可交互的前端 demo

#### Action（行动）—— 重点展开

> **架构设计**: 我采用了 Pipeline + 并行 + 预算循环的三层编排架构。
> Pipeline 层串联 PreferenceAgent 和 DestinationAgent；
> 并行层通过 asyncio.gather 同时执行航班、酒店、活动三个 Agent；
> 预算循环层由 BudgetAgent 校验总花费，超预算时触发渐进式降级。
>
> **并行执行**: 三个搜索 Agent 互相独立，我用 asyncio.gather + return_exceptions=True 实现并行。
> 这样即使某个 Agent 失败，其他 Agent 不受影响。加了 asyncio.wait_for 做超时控制。
>
> **预算循环**: 这是系统最复杂的部分。为避免无限循环，我设计了：
> (1) 最大重试次数限制（3轮）；
> (2) 渐进式降级策略——第1轮砍活动费用，第2轮降酒店等级，第3轮换经济航班；
> (3) 每轮调整幅度按超预算比例动态计算。
>
> **数据模型**: 使用 Pydantic v2 定义了完整的类型系统，
> 包括 TravelPlanState 全局状态对象，确保 Agent 间数据传递类型安全。

#### Result（成果）

> - 行程规划端到端耗时从串行 18s 降至并行 6s，**延迟降低 67%**
> - 预算控制准确率 **98%**（1000 次测试中仅 20 次超出 5%）
> - 代码覆盖率 **85%+**，10 个单元测试全部通过
> - 支持 6 种旅行风格、6 个热门目的地
> - 项目开源在 GitHub，获得同行关注

## 三、面试常见追问与回答策略

### 追问 1："为什么不用单个大 Agent？"

> 单 Agent 会导致 prompt 过长（6000+ tokens），上下文窗口压力大，
> 且不同搜索任务无法并行。拆分为 6 个 Agent 后：
> - 每个 Agent 的 prompt 短小精确（<500 tokens）
> - Flight/Hotel/Activity 可以并行，节省 67% 延迟
> - 职责单一，方便独立迭代和测试

### 追问 2："你遇到的最大挑战是什么？"

> 预算循环的设计。最初我用 while 循环反复调用所有 Agent，
> 但发现两个问题：(1) 可能无限循环；(2) 每轮调整没有递进，
> 总是砍同一个维度。后来我引入了渐进式降级策略和最大轮次限制，
> 让每轮的调整目标不同（活动→酒店→航班），效果显著提升。

### 追问 3："如果要上线生产，还需要做什么？"

> 1. 接入真实 API（Amadeus 航班、Booking 酒店、Google Places 景点）
> 2. 添加 Redis 缓存热门搜索结果
> 3. 接入 Langfuse 做可观测性（Trace 每个 Agent 的耗时和 Token 消耗）
> 4. 用 Kubernetes 部署，支持水平扩展
> 5. 添加用户认证和行程持久化（PostgreSQL）
> 6. A/B 测试不同的推荐算法

### 追问 4："三种语言实现有什么差异？"

> - **Python**: 生态最好（LangGraph/CrewAI），asyncio 原生协程适合 IO 密集
> - **Java**: CompletableFuture 编排复杂但线程安全好，Spring Boot 企业级基础设施完善
> - **Go**: goroutine 并发模型最轻量（创建百万 goroutine 无压力），单二进制部署运维友好

## 四、简历避坑指南

### ❌ 错误示范

```
• 使用 Python 开发了一个旅游规划系统
• 使用了 FastAPI 和 Streamlit
• 实现了多个 Agent
```

**问题**: 太笼统，没有技术深度，没有量化数据

### ✅ 正确示范

```
• 设计 6-Agent 行程规划系统，Pipeline+并行+预算循环三层编排，
  asyncio.gather 并行搜索延迟降低 67%
```

**原则**:
1. **动词开头**: 设计/实现/优化/搭建
2. **技术细节**: 具体用了什么技术、什么模式
3. **量化结果**: 延迟降低 X%、准确率 X%、覆盖 X 个场景
4. **突出亮点**: 预算循环是最大的差异化亮点
