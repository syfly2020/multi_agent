# 多智能体行程规划系统 — 面试问答精编（30+）

每题含：**问题**、**参考答案**（约 200～300 字量级，可按岗位加深）、**面试官想听的点**、**常见答偏/踩坑**。结合本仓库 `TravelPlanningPipeline`、`ParallelExecutor`、`BudgetLoopController` 实现作答更可信。

---

## 一、项目整体设计（8 题）

### Q1. 为什么用多 Agent 而不是单 Agent 一把梭？

**参考答案：**  
单 Agent 要面对「偏好澄清、目的地推理、航班/酒店/活动多源检索、预算仲裁」全部挤在同一上下文里，提示冗长、工具混杂、错误难定位，且无法对独立领域做并行与隔离。多 Agent 把职责拆开：偏好与目的地串行奠基，机酒活动三个搜索彼此无依赖可并行，预算单独做校验与调整策略。编排器掌握控制流，各 Agent 接口统一为 `run(state)`，测试与替换实现（例如换真实机票 API）都更简单。成本上多 Agent 可能增加轮次，但通过共享状态、并行和缓存可以控制；换来的是可维护性与可观测边界清晰。若任务极简单（只答一句话），单 Agent 更合适；本项目的复杂度值得拆分。

**面试官想听的点：**  
职责分离、并行机会、可测试性/可替换性；承认多 Agent 的成本与治理成本。

**常见踩坑：**  
只说「酷/先进」；无法对比单 Agent；忽略共享状态并发写问题。

---

### Q2. 如何设计 Agent 之间的通信？为什么选共享状态而不是消息队列？

**参考答案：**  
当前阶段采用「共享 `TravelPlanState`（Pydantic）+ 显式 Pipeline」：前序 Agent 写入约定字段，后序只读或追加。这样延迟最低、调试直观，适合单体部署与小团队。若未来拆成微服务，可演进为消息信封（带 `correlation_id`、版本号）或事件总线，把 state 的增量作为事件载荷，再配合 Saga 补偿。选型依据是团队规模、一致性要求与是否需要异构语言。消息队列耦合低但运维与观测成本高；共享状态耦合高但实现快。面试中应说明**演进路径**而非教条。

**面试官想听的点：**  
共享状态 vs 消息的 trade-off；幂等、版本、补偿；何时拆分服务。

**常见踩坑：**  
认为只有一种正确答案；忽略 Schema 演进与并发写规范。

---

### Q3. Pipeline、DAG、状态机在本项目里分别体现在哪里？为什么没上 LangGraph？

**参考答案：**  
宏观是 Pipeline：`Preference → Destination → 并行搜索 → 预算`。并行段可看成小型 DAG（三节点无依赖，扇入到 Budget）。`PlanningState` 枚举体现状态机语义（如 `BUDGET_CHECKING`、`ADJUSTING`）。预算调整带循环，严格说不是纯 DAG，需要外层 `BudgetLoopController`。未上 LangGraph 的原因可以是：流程节点数有限、团队希望逻辑全在可读 Python 里、无强需求人机在环与 checkpoint；当分支暴增、要持久化断点或可视化编辑时，再迁移到图框架，且保持 `BaseAgent.run` 接口稳定即可。

**面试官想听的点：**  
能画流程；知道 DAG 与环的区别；框架引入的触发条件。

**常见踩坑：**  
把「用了 Python 函数顺序调用」说成 DAG；说不清何时需要专业编排框架。

---

### Q4. 编排器（Orchestrator）和单个 Agent 的边界怎么划？

**参考答案：**  
编排器负责**控制流**：何时串行、何时并行、何时循环、何时失败短路；不负责具体领域推理。Agent 负责**在既定输入下完成一类子任务**（如只搜酒店），内部可用 LLM、规则或混合。边界清晰的好处是编排器单元测试可 mock Agent，Agent 单测可不启动整条流水线。若编排器里写大量业务规则，会臃肿；若 Agent 里自行决定「下一步调谁」，会难以全局约束（如预算循环次数）。

**面试官想听的点：**  
控制流 vs 领域能力；可测性；防止「智能泄漏」到各处。

**常见踩坑：**  
编排器里堆提示词；Agent 里偷偷调用兄弟 Agent 破坏层次。

---

### Q5. 如何保证「航班/酒店/活动」写入同一份 state 时不互相踩？

**参考答案：**  
约定**字段所有权**：航班只写 `flights` 及相关，酒店只写 `hotels`，活动只写 `activities`；合并后由预算或校验逻辑读全量。更稳妥可改为各 Agent 返回 patch，编排器在 `gather` 后顺序 merge。避免多协程写同一标量（如同时写 `total_cost`）。若未来加锁，需说明 asyncio 下锁的适用场景；多数情况**不共享可变标量**更简单。Code Review 与断言可防回归。

**面试官想听的点：**  
并发写意识、所有权/patch、校验层。

**常见踩坑：**  
以为 asyncio 绝对安全；忽略语义冲突（日期对不上）。

---

### Q6. 系统的「单一事实来源」是什么？用户偏好与中间结果如何版本化？

**参考答案：**  
用户原始偏好以 `UserPreferences` 为入口事实；执行过程中扩展字段挂在 `TravelPlanState`。若支持用户中途改预算，应显式产生新版本或 `revision` 字段，避免 Agent 读到半旧半新。持久化时可存 `state` 快照 + `updated_at`。对审计场景，记录每次 Agent 输出的 hash 或 JSON 片段有助于追责。

**面试官想听的点：**  
状态快照、修订策略、审计需求。

**常见踩坑：**  
全局可变 dict 到处传、无版本概念。

---

### Q7. 若要把本项目做成 SaaS，架构上会改哪些？

**参考答案：**  
API 层无状态，会话状态入 Redis/DB；Agent 执行可放队列 worker，支持异步任务与 webhook 回调；工具层加统一鉴权、限流、熔断与缓存；多租户隔离数据与密钥；观测上 OpenTelemetry + 按租户配额。编排可仍用 Python，或抽成独立「规划服务」。LLM Key 走服务端，不把密钥下发浏览器。

**面试官想听的点：**  
无状态、队列、多租户、安全、可观测。

**常见踩坑：**  
只谈「上 K8s」不谈状态与配额。

---

### Q8. 多 Agent 系统如何做集成测试与端到端测试？

**参考答案：**  
单元测试 mock LLM 与外部 API，断言 state 字段变化；编排器测试注入 fake Agent（固定返回）；契约测试校验 `TravelPlanState` Schema。E2E 可用录制的 tool fixture 或小范围沙箱 API。对非确定性输出，断言结构约束与不变量（预算字段存在、日期合法）而非逐字比对。CI 里控制真实 LLM 调用次数（夜间或手动闸门）。

**面试官想听的点：**  
分层测试、契约、非确定性处理。

**常见踩坑：**  
全靠人工点 UI；E2E 强依赖付费 API 导致 CI 不稳定。

## 二、并行与性能（6 题）

### Q9. 并行 Agent 里某一个出错怎么办？会不会拖垮整次规划？

**参考答案：**  
`asyncio.gather(..., return_exceptions=True)` 收集异常，记录到 `error_messages`，其余 Agent 结果仍保留。下游预算 Agent 应能处理部分缺失：例如无活动时按零或给默认，无航班则可能判 `FAILED`（关键路径）。关键与非关键要有分级。对用户返回需汇总哪些维度失败、是否可重试。日志要带 Agent 名与异常栈。

**面试官想听的点：**  
部分失败模型、关键路径、用户体验。

**常见踩坑：**  
一律失败或一律忽略错误。

---

### Q10. 为什么用 `asyncio.gather` 而不是 `ThreadPoolExecutor`？

**参考答案：**  
子 Agent 主要是 IO 等待（HTTP 调工具、LLM API），`asyncio` 在单线程内协作式切换，开销小、易组合超时 `wait_for`。线程池适合**阻塞式** SDK 或 CPU 密集计算。若某工具只有同步客户端，可用 `to_thread` 包一层，但要注意线程数上限。CPython GIL 下 CPU 密集仍应用多进程或原生扩展。

**面试官想听的点：**  
IO vs CPU；GIL 意识；阻塞代码的桥接。

**常见踩坑：**  
在 async 函数里直接调阻塞 IO 导致事件循环卡死。

---

### Q11. 如何给并行阶段设超时？超时后策略是什么？

**参考答案：**  
对每个 `agent.run` 包 `asyncio.wait_for(..., timeout=PARALLEL_TIMEOUT)`，超时触发 `TimeoutError`，记入错误信息。策略可选：该维度降级为空、用缓存结果、或整次规划失败（若关键）。超时阈值应略大于 P99 外部依赖，并配合熔断。监控上单独统计「并行段最长 span」。

**面试官想听的点：**  
`wait_for`、降级 vs 失败、监控。

**常见踩坑：**  
只设总超时不设单 Agent 超时，无法定位慢在哪。

---

### Q12. 如何监控并行执行是否变慢、哪一维最拖后腿？

**参考答案：**  
为 `ParallelExecutor.run` 建 trace span；每个子 Agent 子 span；记录耗时、是否超时、是否异常。Metrics：`parallel_wall_ms`、`agent_x_p95_ms`。日志里打 `attempt` 轮次。结合外部 API 的 HTTP 指标区分「模型慢」还是「供应商慢」。

**面试官想听的点：**  
分布式追踪、分位数、分解瓶颈。

**常见踩坑：**  
只记整请求耗时无法下钻。

---

### Q13. 并行会不会打爆外部 API QPS？如何限流？

**参考答案：**  
在工具层用令牌桶/信号量限制并发；按供应商分区配额；缓存重复查询；合并批量接口若存在。多用户场景在网关按租户限流。预算循环每轮都并行时，总 QPS = 并发用户 × 每轮 Agent 数 × 轮次风险，需乘算进容量规划。

**面试官想听的点：**  
全局限流、缓存、容量公式意识。

**常见踩坑：**  
只考虑单用户 demo。

---

### Q14. 若三个 Agent 都要调同一个 LLM，并行会不会触发速率限制？

**参考答案：**  
会。需在 LLM 客户端侧共享限流器（全局 semaphore + 速率），或串行化 LLM 调用而仅并行工具 HTTP。另一策略是批处理或换具备更高配额的部署。观测 `429` 与 retry-after。面试中说明**并行的是 IO 不是无节制并行模型调用**。

**面试官想听的点：**  
429、全局限流、架构取舍。

**常见踩坑：**  
假设云厂商配额无限。

---

## 三、预算循环（5 题）

### Q15. 如何避免预算调整「无限循环」？

**参考答案：**  
必须同时具备：**最大轮次**（如 `BUDGET_MAX_RETRIES`）、**成功/失败明确终止条件**、可选**震荡检测**（费用几乎不变仍不达标则停）。禁止仅依赖模型口头「再试一次」。每轮调整后应改变策略（渐进式降级），否则易来回抖。系统级还可加 Token/总耗时上限。

**面试官想听的点：**  
硬上限、震荡、策略单调性。

**常见踩坑：**  
只信任 LLM 自停。

---

### Q16. 降级策略的优先级应怎么定？

**参考答案：**  
先动对体验伤害小、弹性大的部分：削减可选活动、降低酒店档位、最后动航班（改签成本高、用户敏感）。也可按「成本弹性」排序：活动 > 酒店 > 机票。需产品一致认可并写进规则或提示，避免模型随意发挥。每轮降级强度随 `attempt` 递增。

**面试官想听的点：**  
业务优先级、可解释性、轮次递增。

**常见踩坑：**  
没有明确优先级，全靠模型猜。

---

### Q17. 预算校验的「精度」问题怎么处理？浮点误差和币种呢？

**参考答案：**  
金额用 `Decimal` 或在展示层四舍五入、比较时用容忍带 `epsilon`（如 ±1 元）。币种统一在 Schema 层（本项目人民币）避免混算。税费、汇率若引入需单独字段与快照时间。对外展示与内部计算精度可分离。

**面试官想听的点：**  
Decimal/epsilon、一致币种、汇率快照。

**常见踩坑：**  
直接 `==` 比较 float。

---

### Q18. 达到最大重试仍未满足预算，应该返回什么？状态码怎么设计？

**参考答案：**  
应返回**当前最优方案** + **明确警告**（超预算金额、哪些维度已降级），让用户二次决策。内部状态可用 `COMPLETED` + `warnings` 或引入 `PARTIAL`。HTTP API 可 200 + 业务字段 `satisfied_budget: false`，或 422 视产品约定；关键是**前端可渲染、可重试**。日志标记 `budget_exhausted`。

**面试官想听的点：**  
best-effort、契约清晰、勿静默失败。

**常见踩坑：**  
与「成功」语义混淆，用户以为在预算内。

---

### Q19. 预算循环和 LangGraph 的 conditional edge 怎么对应？

**参考答案：**  
预算通过则边到结束；不通过则边回「调整+并行搜索」节点，形成环。`max_retries` 对应图上的计数器或 `remaining_steps`。LangGraph 可把状态持久化，本项目是内存循环，语义等价。迁移时把 `BudgetLoopController` 逻辑拆成节点函数即可。

**面试官想听的点：**  
图与循环、持久化差异。

**常见踩坑：**  
说不清环与 DAG 区别。

---

## 四、工程化（6 题）

### Q20. 如何保证多 Agent 系统的可观测性？

**参考答案：**  
OpenTelemetry trace 贯穿 Pipeline、各 Agent、工具 HTTP；span 属性带 `attempt`、`planning_state`、模型名；日志带 `trace_id`。Metrics：延迟分位、错误率、每请求 LLM 次数、预算轮次分布。对用户支持暴露 `request_id` 查询。Prompt/Completion 采样存对象存储需注意脱敏与合规。

**面试官想听的点：**  
Trace/Span/Metrics 三角；PII；采样。

**常见踩坑：**  
只有 print 没有关联 id。

---

### Q21. 如何做灰度发布与配置开关？

**参考答案：**  
新 Agent 实现挂 feature flag（按用户百分比、白名单）；默认走旧路径。配置 `max_retries`、超时、模型名走配置中心，避免改代码发版。金丝雀发布规划服务实例，对比错误率与延迟。回滚策略预先演练。

**面试官想听的点：**  
Feature flag、配置外置、金丝雀。

**常见踩坑：**  
大开关一刀切无观测。

---

### Q22. 「百万并发」在本场景下怎么理解？如何设计？

**参考答案：**  
规划请求重 IO、重 LLM，百万 QPS 不现实；应转化为**百万用户规模下峰值规划 QPS** 与**异步化**。接入层水平扩展；规划任务入队；worker 池消费；依赖 LLM/API 的配额与多区域部署。状态存 Redis/DB，避免进程内全局。同步 API 仅接受任务 id，结果轮询或推送。

**面试官想听的点：**  
异步队列、限流、水平扩展、现实峰值。

**常见踩坑：**  
空谈百万并发无队列与背压。

---

### Q23. 密钥与第三方 API 如何管理？

**参考答案：**  
环境变量或密钥管理服务（Vault、云 KMS）；绝不下发前端。按环境分 key；轮换与最小权限。工具层统一封装重试与审计日志。防止日志打印完整 API Key。

**面试官想听的点：**  
KMS、轮换、最小权限。

**常见踩坑：**  
密钥进仓库。

---

### Q24. Schema（Pydantic）变更如何兼容线上？

**参考答案：**  
新增字段默认可选；废弃字段保留只读一段时间；版本号 `schema_version` 或在 API 路径体现。双写双读过渡期。契约测试防止 Agent 输出缺字段。迁移时回放历史 state 样本做兼容测试。

**面试官想听的点：**  
向后兼容、契约测试、演进策略。

**常见踩坑：**  
破坏性改字段导致线上反序列化失败。

---

### Q25. 如何做容量规划与成本仪表盘？

**参考答案：**  
估算：日均规划次数 × 每规划 LLM token × 单价 + 外部 API 调用费。监控实际 token 与轮次分布，发现异常爬升。按租户/功能拆解成本，驱动是否引入小模型或缓存。预算告警接财务流。

**面试官想听的点：**  
公式化估算、监控驱动优化。

**常见踩坑：**  
上线后才发现成本不可接受。

---

## 五、LLM 相关（5 题）

### Q26. Token 消耗怎么优化又不伤质量？

**参考答案：**  
裁剪 state 只传必要字段；系统提示模板化；简单子任务路由小模型；工具结果截断与摘要；缓存重复检索；减少无效循环轮次；结构化输出减少废话。质量用离线集与线上抽检监控，避免单看 token。

**面试官想听的点：**  
上下文工程、路由、缓存、评估。

**常见踩坑：**  
一味删提示导致违规或胡编增加。

---

### Q27. Prompt 设计上有哪些原则？

**参考答案：**  
角色与目标清晰；输出格式用 JSON Schema 约束；Few-shot 选高质少量；明确不可为（如不得编造价格）；工具描述与参数名自解释；链式步骤拆到不同 Agent 比单提示堆砌更有效。对中文用户注意日期与货币单位说明。

**面试官想听的点：**  
结构化、边界、示例质量、分步。

**常见踩坑：**  
提示无限加长；无格式约束。

---

### Q28. 如何降低幻觉（尤其价格、航班信息）？

**参考答案：**  
价格、库存等**必须以工具为准**，提示中写明「不得臆造工具未返回的数据」；工具失败要明确返回错误对象；可加重查（Reflection）只对结构化字段校验。对用户展示带来源与时间。RAG 文档若与实时 API 冲突，以 API 为准。

**面试官想听的点：**  
工具权威、失败显式、校验层。

**常见踩坑：**  
允许模型在没调用工具时编价格。

## 六、Java / Go 实现差异（4 题）

### Q31. 若用 Java，并行编排用 `CompletableFuture` 怎么对应 `asyncio.gather`？

**参考答案：**  
`CompletableFuture.allOf(f1, f2, f3).join()` 类似 gather；单个超时用 `orTimeout`；异常用 `handle` 或 `exceptionally` 实现「部分失败继续」。注意默认 `ForkJoinPool` 与自定义线程池隔离阻塞调用。Spring WebFlux 则更接近响应式链。与 Python 一样要明确**共享状态写入规则**。

**面试官想听的点：**  
allOf、超时、异常处理、线程池。

**常见踩坑：**  
阻塞 IO 放在默认 common pool。

---

### Q32. 若用 Go，为什么可能用 goroutine + channel 而不是纯共享 struct？

**参考答案：**  
Go 鼓励「通过通信共享内存」：每个 Agent goroutine 返回结果到 channel，主 goroutine merge，避免锁复杂化。也可 `errgroup` 收集错误。共享指针 struct 需注意并发写，可用 mutex 或 copy-on-write。上下文用 `context.Context` 做超时与取消，对应 asyncio 的 cancel 语义。

**面试官想听的点：**  
channel/errgroup、Context、并发安全。

**常见踩坑：**  
多 goroutine 写同一 map 无保护。

---

### Q33. Python / Java / Go 在本项目中如何分工（若三语言共存）？

**参考答案：**  
常见分层：Go 做高并发 API 网关与接入；Java 做与企业遗留集成（票务、ERP）；Python 做 LLM Agent 与快速迭代的研究侧。通过 gRPC/JSON 消息传递 `TravelPlanState` 序列化。代价是运维与类型契约成本上升，仅在中大型团队值得。小团队优先单语言单体。

**面试官想听的点：**  
按瓶颈与生态分工、契约、团队成本。

**常见踩坑：**  
为技术栈而技术栈，忽略维护负担。

---

### Q34. Java/Go 里如何实现「预算循环」与「可观测」？

**参考答案：**  
循环仍是 for + 计数器 + 枚举状态；用配置注入 `maxRetries`。观测用 Micrometer/OpenTelemetry（Java）或 OTel Go SDK，创建 span 层级与 Python 一致。日志用结构化 JSON（Logback/zerolog）带 trace id。关键是在两种语言里保持**同一 Trace 上下文传播**（W3C traceparent）。

**面试官想听的点：**  
状态机不变、OTel 统一、跨语言 trace。

**常见踩坑：**  
各语言各打各的日志无关联 id。
