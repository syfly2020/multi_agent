## 1. Agent vs Workflow vs Multi-Agent

**Agent（智能体）** 强调在环境中「感知—推理—行动」的闭环：观测可以是用户一句话、工具返回的 JSON、上一轮的中间状态；行动可以是自然语言回复、一次 Function Call、写入内存。与普通过程式代码的关键差异是：**下一步由模型在运行时决定**，而不是在开发期写死 `if/else` 的完整树。因此 Agent 适合开放域、需求表述模糊、需要多轮澄清或多工具组合的任务，例如「帮我规划一趟兼顾亲子和预算的行程」。

**Workflow（工作流）** 强调控制流在编排层预先定义：步骤 A 完成后进入 B，条件分支由业务规则或配置表决定。LLM 可以嵌入在某一步做「内容生成」或「分类」，但**谁调用谁、失败往哪跳**通常不由模型即兴发挥。适合强合规、可审计、步骤稳定的场景（财务审批、工单 SLA、ETL）。优点是行为可预测、易做集成测试；缺点是遇到未预料分支时要改编排而不是靠模型「临场发挥」。

**Multi-Agent（多智能体）** 指多个（逻辑或部署上的）Agent 协作完成目标：可以角色分工（研究员/写手/审查员），可以并行子任务（航班/酒店/活动同时搜），也可以互相发消息辩论。与「单 Agent + 很多工具」的边界在于：是否显式存在**多个上下文边界、多个目标函数、或需要并行合并与冲突仲裁**。本项目采用 `TravelPlanningPipeline` 串起 `PreferenceAgent`、`DestinationAgent`，再经 `ParallelExecutor` 并行三个搜索 Agent，最后由 `BudgetAgent` 做预算校验与调整建议——这是典型的**编排器主导 + 多子 Agent 执行**，而不是完全对话驱动的自由多 Agent。

**何时用哪种**：步骤清晰且少变 → Workflow；单一路径但需要灵活工具链 → 单 Agent；子任务可并行、需领域拆分或互相校验 → Multi-Agent。**误区**：把「class 里写三个方法」当成三个 Agent——若没有独立提示、独立工具策略或并行语义，通常只是代码模块化。

```python
# Workflow：控制流在开发者手中
def travel_workflow(prefs):
    s = preference_step(prefs)
    s = destination_step(s)
    return search_step(s)

# Agent：控制流部分交给模型（示意循环）
# observation = initial_state
# while not done:
#     plan = llm.plan_next_step(observation)
#     observation = execute(plan)  # 可能含 tool calls
```

**面试追问**：「多 Agent 会不会更贵？」——通常会增加轮次与上下文重复，需用共享状态、并行与缓存控制成本。

## 2. Pipeline / DAG / State Machine

**Pipeline（流水线）** 把端到端过程看成有序阶段，数据像流水一样经过每一站：入口统一、出口统一，中间阶段依次执行。优点是心智模型简单、日志按阶段打即可对齐排障、单元测试可以「喂一个伪造 state，只测某一阶段」。缺点是当出现「预算不通过要回到搜索阶段」这类**回边**时，纯线性 Pipeline 需要额外的循环控制器（如本项目的 `BudgetLoopController`），否则代码里会散落多个 `while` 与标志位。

**DAG（有向无环图）** 用节点表示任务、用边表示依赖关系；无环保证存在拓扑排序，可最大化并行（无依赖的节点同时跑）。适合构建系统、特征工程等多源依赖场景。严格 DAG **不允许环**，而「预算—再搜索—再预算」是环，因此工程上常见做法是：**DAG 负责主依赖**，环由外层「控制循环」或「超级节点」封装，或改用支持条件边的图框架（如 LangGraph）。

**State Machine（状态机）** 用枚举状态与显式转移描述生命周期，例如 `PlanningState.COLLECTING_PREFERENCES` → `RECOMMENDING_DESTINATIONS` → `BUDGET_CHECKING` → `ADJUSTING`。好处是产品、测试、监控都能用同一套状态语言沟通；持久化后可做断点续跑。LangGraph 等框架本质是把「状态 + 图上的路由函数」结合在一起，比手写 if 链更易维护。

**三者对比小结**：Pipeline 胜在简单；DAG 胜在依赖表达与并行调度；状态机胜在可观测与合规。**本项目**：宏观是 Pipeline；并行段是「扇出—扇入」；预算调整是带终止条件的循环，语义上对应状态机中的 `ADJUSTING` 回路。

```text
Preference → Destination → ┌ Flight  ┐
                          │ Hotel   │ → Budget ⇄ (未通过则调整后再并行)
                          └ Activity┘
```

**面试追问**：「为什么不用纯 DAG 描述全流程？」——因为存在条件循环与业务终止策略，需要显式上限与失败语义，纯 DAG 表达吃力。

## 3. ReAct / CoT / Reflection

**CoT（Chain-of-Thought）** 通过提示语让模型输出中间推理步骤（「先分析…再比较…因此…」），再给出最终答案。对数学、逻辑、多约束取舍类问题往往有效；不依赖外部环境，一次或少数几次补全即可完成。缺点是上下文变长、若步骤错误会「一本正经延续错误」；适合嵌入在单个 Agent 的某次 `generate` 里，例如目的地推荐理由的结构化推导。

**ReAct** 将推理与行动交织：`Thought` 说明意图，`Action` 调用工具，`Observation` 把结果接回上下文，循环直到任务完成。它是现代「工具调用 Agent」的理论骨架之一，与 OpenAI 的 `tool_calls` 消息流一致。适合需要查实时价、天气、库存的行程规划；本项目的各搜索 Agent 内部若采用「模型决定调用哪个 search 函数」即是 ReAct 风格。

**Reflection** 在初稿产出后增加「评审—修订」环节：同一模型换系统提示扮演 critic，或换更强模型做审查。可显著减少事实错误与格式错误，但增加延迟与 Token。典型模式：Generator 输出行程 JSON → Critic 列出违反约束项 → Generator 只改违规部分。

**对比表（记忆用）**：

| 模式        | 是否常用工具 | 典型代价     | 适用环节           |
|-------------|--------------|--------------|--------------------|
| CoT         | 否           | 中           | 单次推理加深       |
| ReAct       | 是           | 中高         | 检索与实时数据     |
| Reflection  | 可选         | 高           | 交付前质量门       |

```text
ReAct 轨迹片段:
Thought: 用户预算 8000，需先查往返机票区间。
Action: search_flights(origin, dest, dates)
Observation: lowest=3200 ...
Thought: 剩余预算可分配给酒店与活动。
```

**面试追问**：「ReAct 和 Function Calling 什么关系？」——Function Calling 是协议与运行时；ReAct 是提示与推理样式，常一起用。

## 4. Tool Calling / Function Calling

**工作原理**：大模型在补全阶段不直接执行代码，而是输出符合接口约定的**调用描述**（函数标识符 + 参数 JSON）。宿主应用解析后执行真实函数（HTTP、DB、本地计算），再把返回值封装为 `role=tool`（或历史里的 `function`）消息追加到对话，模型基于新观测继续生成。这样把「不可信生成」与「可信执行」分离：执行前可做权限校验、参数范围校验、限流。

**协议细节（OpenAI 兼容常见形态）**：

1. **工具清单**：请求体中带 `tools: [{type:"function", function:{name, description, parameters}}]`，`parameters` 多为 JSON Schema。
2. **模型响应**：`assistant` 消息含 `tool_calls` 数组，每项有 `id`、`function.name`、`function.arguments`（字符串化 JSON）。
3. **结果回传**：每条 tool 结果对应 `tool_call_id`，顺序可以交错但 id 必须对齐。
4. **并行**：同一轮多个 `tool_calls` 可由 `asyncio.gather` 并行执行，注意线程安全与超时。
5. **错误语义**：函数抛错应编码进返回内容（如 `{"error":"timeout"}`），否则模型易幻觉成功。

**与多 Agent 关系**：工具是「能力扩展」；Agent 是「决策边界」。一个 Agent 可暴露多个工具；编排器也可绕过 LLM 直接调工具（纯代码路径），用于降低成本。

```python
# 消息序列示意（简化）
msgs = [
  {"role": "user", "content": "查北京到上海明天最低价机票"},
  {"role": "assistant", "tool_calls": [{
      "id": "call_1", "type": "function",
      "function": {"name": "search_flights", "arguments": '{"from":"PEK","to":"SHA"}'}
  }]},
  {"role": "tool", "tool_call_id": "call_1", "content": '{"lowest": 980}'},
]
```

**面试追问**：「如何防止模型乱调工具？」——Schema 约束、允许列表、服务端二次校验、对危险操作要人工确认或二次认证。

## 5. 并行 Agent 的状态合并与冲突解决

**问题来源**：`asyncio.gather` 并发执行多个 `agent.run(state)` 时，若多个协程同时修改同一字典或同一 Pydantic 对象字段，可能出现丢失更新或数据结构不一致。Python 协程虽非多线程抢占，但在 `await` 边界仍可能交错写入共享对象。

**策略一：分片写入（Field Ownership）**：约定 `FlightAgent` 只写 `state.flights` 及相关字段，`HotelAgent` 只写 `state.hotels`。字段不交叠则无需锁。本项目若严格执行该约定，则与「多协程共享一个 state 引用」兼容。**风险**：新人改代码时破坏约定，需 Code Review 或运行时断言。

**策略二：Reducer / Patch**：各 Agent 不直接改全局 state，而是返回 `FlightPatch` 等不可变增量，主线程在 `gather` 完成后**顺序** `apply`。合并顺序若影响结果（例如都改 `total_cost`），需定义优先级或合并函数。

**策略三：深拷贝输入**：每个 Agent 拿到 `state.model_copy(deep=True)`，只回传自己负责的部分，由编排器 merge。内存与序列化成本更高，但并发最安全。

**冲突类型与解决**：(1) **数据层双写**：用所有权或 patch；(2) **语义矛盾**：如酒店入住日晚于离境日——由校验 Agent 或规则引擎在合并后跑一遍；(3) **部分失败**：本项目用 `return_exceptions=True` 把异常记入 `error_messages`，不使整个 gather 失败，属于**部分成功模型**，下游需判断关键字段是否缺失。

```python
# Patch 合并伪代码
async def safe_parallel(agents, state):
    copies = [state.model_copy(deep=True) for _ in agents]
    parts = await asyncio.gather(*(a.run(c) for a, c in zip(agents, copies)))
    return merge(state, parts)  # 显式 merge 规则
```

**面试追问**：「为什么不用锁？」——在 asyncio 单线程内，锁主要防的是显式 `create_task` 中的重入与复杂交错；更简单的是不共享可变状态。

## 6. 预算循环的终止条件设计

**业务目标**：在用户预算约束下给出可执行行程，同时在超预算时能自动降级（减活动、降酒店星级等），但不能让用户无限等待，也不能让后端 Token 与外部 API 调用失控。

**终止条件清单（推荐全覆盖）**：

1. **成功终止**：`BudgetAgent` 计算总费用 ≤ 用户预算（加容忍带），`state` 置 `COMPLETED`。
2. **计数终止**：`max_retries` / `max_adjustments` 用尽即停，防止无限循环；本项目 `BudgetLoopController` 使用 `for attempt in range(self.max_retries + 1)` 即此类硬上限。
3. **失败终止**：出现不可恢复错误（如无航班、签证不符、用户偏好互斥）置 `FAILED`，**不再重试**以免浪费资源。
4. **震荡终止（增强项）**：若连续 k 轮总成本变化 < ε 仍超预算，判定策略无效，返回当前最优并文字说明「无法在约束内完成」。
5. **资源终止**：全局 API 超时、LLM 单次/累计 Token 上限、钱包余额告警——属于平台级熔断。

**与产品文案一致**：计数用尽后当前实现将状态标为 `COMPLETED`（可讨论是否改为 `PARTIAL`），面试中应说明**刻意选择**：避免前端无结果，但需在响应体里带 `warnings` 说明未严格满足预算。

**渐进式降级**：按「对体验伤害小→大」排序调整维度（例如先砍可选活动，再换酒店，最后动航班），减少来回震荡。**反模式**：仅依赖模型输出「要不要再试」而无计数器——极易死循环。

```python
# 终止条件逻辑骨架（示意）
for attempt in range(max_rounds):
    state = await parallel_search(state)
    state = await budget_check(state)
    if within_budget(state):
        return finalize(state)
    if unrecoverable(state):
        return fail(state)
    state = adjust_strategy(state, attempt)  # 轮次越高，降级越强
return best_effort(state)
```

## 7. Agent 间通信机制

**消息传递（Message Passing）**：每个 Agent 有收件箱（队列），消息体包含 `correlation_id`、`payload`、`reply_to`。优点：解耦、易横向扩展为多进程/多机；缺点：需序列化 Schema、处理重复投递与顺序。**适用**：微服务拆分、异构语言（Go/Java/Python）混部。

**共享状态（Shared State）**：所有 Agent 读写同一结构化对象（如 `TravelPlanState`）或同一数据库行版本。**优点**：实现快、调试直观；**缺点**：并发要写规范、Schema 迁移要协调。**适用**：单体或小型服务，与本项目当前结构一致。

**事件总线（Event Bus）**：Agent 发布事件（`DestinationChosen`），订阅者异步响应。优点：极松耦合、易插拔新能力；缺点：全链路追踪要额外建设（Trace、事件溯源）。**适用**：插件市场、多团队共建平台。

**对比与选型**：

| 机制     | 耦合度 | 调试难度 | 扩展性 |
|----------|--------|----------|--------|
| 共享状态 | 高     | 低       | 中     |
| 消息     | 低     | 中       | 高     |
| 事件总线 | 很低   | 高       | 很高   |

**Saga 补充**：长事务跨多个 Agent/服务时，需补偿动作（订酒店失败则释放航班 hold）；通信机制无论选哪种，都要设计**幂等与补偿**。

## 8. LangGraph vs CrewAI vs AutoGen（2026 视角）

**LangGraph**：以**有状态图**为核心，节点是函数或 Agent，边可有条件；支持 checkpoint、人机在环、时间旅行调试。与 LangChain 工具链集成深。适合要把「预算循环 + 并行分支」做成可持久化工作流的团队。**缺点**：概念多，小项目有学习成本。

**CrewAI**：以 **Crew / Agent / Task** 组织角色与任务，YAML/代码描述「谁负责什么」，上手快、演示效果好。适合营销文案、研究报告等多角色协作原型。**缺点**：强约束生产管线时，细粒度控制与观测有时需自己补。

**AutoGen（及微软生态演进）**：强调**对话式**多 Agent（GroupChat、selector），灵活度高，适合研究与探索式任务。**缺点**：流程「飘」、工程化约束要自建状态机或外层编排。

**选型建议**：要强状态与审计 → LangGraph；要快演示角色分工 → CrewAI；要对话博弈与实验 → AutoGen。本项目用原生 asyncio 实现 Pipeline，面试中可答：**流量与复杂度未到框架阈值时，手写编排更透明；未来可迁移到 LangGraph 而不改 Agent 接口。**

## 9. 可观测性（Trace / Span / Metrics）

**Trace**：标识一次端到端请求（如 `trace_id=abc`），串联用户入口、编排器、各 Agent、每次 LLM 调用、每次 HTTP 工具调用。**Span**：Trace 内的片段，记录开始时间、结束时间、父 span、属性（`model=gpt-4o`, `attempt=2`）、事件（如「开始流式输出」）。**Metrics**：聚合维度——QPS、P95 延迟、错误率、每 Trace 的 LLM 调用次数、预算循环平均轮数、工具超时率。

**落地步骤**：(1) 用 OpenTelemetry SDK 在 `TravelPlanningPipeline.run` 建 root span；(2) 每个 `agent.run` 建 child span；(3) 工具层建嵌套 span；(4) 日志里注入 `trace_id`；(5) 导出到 Jaeger/Tempo/Grafana Tempo + Prometheus。

**业务与技术指标联动**：在 span attribute 里写 `planning_state`（注意 PII），排障时从「用户说卡住」直接过滤 `PlanningState.ADJUSTING` 的 trace。**面试亮点**：说明白「为什么 Agent 系统比 CRUD 更难观测」——非确定性、多轮、工具失败与模型重试交织。

```python
# 伪代码：Span 嵌套
with tracer.start_as_current_span("TravelPlanningPipeline.run") as root:
    with tracer.start_as_current_span("ParallelExecutor.run"):
        ...
```

## 10. Token 消耗优化与成本控制

**输入侧**：裁剪 `TravelPlanState` 只传当前 Agent 需要的字段；对长对话做**滑动窗口 + 摘要**；系统提示固定模板化，避免重复冗长说明。**输出侧**：用 JSON Schema / `response_format` 约束字段，减少废话；区分「对内推理」与「对用户展示」，避免把长 CoT 暴露给用户又占日志。

**调用侧**：简单任务路由到小模型；缓存 embedding 与检索结果；相同航班查询短期 TTL 缓存；合并工具调用减少往返。**循环侧**：预算循环硬上限；检测重复 query 打断。

**计费与配额**：记录每次调用的 `prompt_tokens`、`completion_tokens`，按 `user_id` 聚合；对接预算告警。产品层可做「每日规划次数上限」。

**面试表述**：优化不是「一味压提示」，而是**在质量、延迟、成本三角**里用可观测数据驱动；AB 测试不同提示长度对转化率与投诉率的影响。

## 11. MCP（Model Context Protocol）

**定义**：MCP 是连接 AI 应用与外部「上下文源」的开放协议：服务端暴露 **Tools**（可执行）、**Resources**（可读 URI 资源）、**Prompts**（模板）等原语，客户端（IDE、Agent 宿主）按标准握手发现能力并调用。

**重要性**：(1) **避免 N×M 适配**：每个模型 × 每个数据源各写一套胶水层不可持续；(2) **安全边界清晰**：由 MCP Server 控制鉴权与数据范围；(3) **生态复用**：天气、地图、内部 Wiki 可一次封装多处使用。

**与 Function Calling**：MCP 偏**服务发现 + 能力目录**；一次具体调用仍常映射为模型侧的 tool call。**关系**：MCP Server 可视为「工具的远程注册中心」。

**本项目联想**：把 `tools/flight_search.py` 等封装为标准 MCP Server，则 CLI、Cursor、内部运营后台可共用同一实现；编排层 Agent 不变，仅工具接入方式改变。

## 12. A2A（Agent-to-Agent Protocol）

**定位**：Google 等推动的**跨系统 Agent 互操作**方向（规范与实现仍在快速迭代），关注 Agent 如何声明能力卡片（Agent Card）、如何委托任务、如何异步拉取结果、如何做身份验证，使不同厂商的 Agent 能组成**联邦工作流**。

**与 MCP 区分**：MCP 让模型获得「上下文与工具」；A2A 让 Agent 作为**对等端**互相委派任务。**典型场景**：企业采购 Agent 调用外部差旅 Agent；本平台 Agent 调用合作伙伴「当地向导」Agent。

**工程注意**：委托链路易产生**责任边界模糊**与**数据泄露**；需 OAuth/mTLS、审计日志、幂等任务 id；对关键写操作保留人工审批节点。

**面试话术**：「我了解 A2A 解决的是生态互连；落地前会评估安全与 SLA，短期内单体系统用共享状态即可。」

## 13. RAG 与 Agent 的结合

**RAG** 通过检索把相关文档片段注入上下文，再生成回答，缓解「训练截止知识」与「私有数据」问题。**Agent + RAG** 常见形态：(1) **工具型**：`retrieve_docs(query)` 作为一个 tool，由 ReAct 决定是否检索；(2) **管线型**：先 RAG 再规划，检索结果写入 state 供后续 Agent 只读；(3) **Agentic RAG**：模型根据上轮答案改写 query、多跳检索、交叉验证。

**行程场景**：目的地签证政策、季节性闭馆信息、酒店取消规则等适合向量库 + 元数据过滤；**实时价格**仍应以工具 API 为准，不能用 RAG 文档代替。

**质量要点**：chunk 策略、hybrid 检索、重排序、引用出处（便于合规）；评估用 golden set 看「检索命中率 + 最终回答正确率」。

```text
User → Planner Agent → [retrieve("日本签证 2026")] → state.docs
     → Destination Agent 结合 docs 与工具结果输出
```

## 14. Agent Memory 系统

**短期记忆（Working Memory）**：当前会话消息、`TravelPlanState`、最近工具结果。实现：token 超限时对早期轮次做**摘要**（可再开一个廉价模型调用），保留关键约束（预算、日期、忌口）结构化字段不动。

**长期记忆（Long-term）**：跨会话稳定偏好（常住城市、舱位偏好、无障碍需求）。实现：用户画像表 + 可选向量检索；写入需用户同意与合规审查。

**情景记忆（Episodic）**：单次行程的决策轨迹与用户反馈（「酒店太远」）。实现：按 `trip_id` 存 JSON + 向量，用于「像上次一样」类请求；检索时用时间衰减。

**陷阱**：不加过滤地把所有历史塞进 prompt → 噪声大、隐私风险、成本高。**最佳实践**：结构化字段优先，非结构化记忆检索 top-k + 分数阈值；重要约束用**显式字段**而非仅靠自然语言记忆。

## 15. 多 Agent 系统的容错与降级

**容错层次**：(1) **任务级**：`gather(..., return_exceptions=True)`，单 Agent 失败记录错误，其他结果保留；(2) **步骤级**：前序 `FAILED` 则短路不跑后续（本项目 Pipeline 在 `Preference`/`Destination` 失败后 return）；(3) **系统级**：熔断下游 API、切换备用模型。

**降级策略**：工具超时 → 读缓存或返回「该维度暂不可用」占位；模型超时 → 规则模板生成骨架行程；预算无法满足 → 返回 best-effort + 明确提示。**优先级**：安全与合规 > 核心路径（机酒）> 增值项（活动推荐）。

**重试**：仅对幂等 GET 类；POST 预订类要幂等键；指数退避 + 抖动防惊群。

**与本项目**：`error_messages` 累积可观测；预算轮次用尽后的语义建议在面试中主动讨论是否引入 `PARTIAL` 与 `warnings` 字段，体现**产品化思维**。
